AI 核心技术
LLM（Large Language Model）大型语言模型
项目使用三个供应商：GROQ（llama-3.3-70b）、Google Gemini 2.0 Flash、Anthropic Claude Haiku。每日处理数百篇 AI 领域资讯，输出结构化分析报告。
Prompt Engineering（提示词工程）
为 PM / 开发者 / 工程师 / 运营四种角色设计专属 System Prompt，引导 LLM 从不同视角提炼资讯价值。同一批内容产出四份差异化报告。
Multi-provider Fallback Chain（多模型降级链）
GROQ 免费套餐每日 100k Token 上限，Anthropic 按量付费，Gemini 有每日请求配额。三者同时触碰限制时，系统自动切换：GROQ → Gemini → Anthropic，保障服务连续性。
TTS（Text-to-Speech）+ AWS Polly 神经语音
LLM 将报告提炼为口语化播客脚本，再由 AWS Polly Zhiyu 中文神经语音（Neural Engine）合成 MP3。Neural 引擎基于深度学习，比传统 TTS 更接近真人语气。
Token 配额管理
Token 是 LLM 的计量单位，约 1 个中文字 = 1.5~2 tokens。4 角色分析 × 多维度 = 12+ 次 API 调用，需严格控制每次请求的输入长度（每角色内容截断为 400 字符）以避免超限。
OpenAI-Compatible API 统一接口
GROQ 和 Gemini 均提供兼容 OpenAI 格式的 API，项目用同一套 OpenAI SDK 接入不同供应商，只需切换 baseURL 和 apiKey，大幅降低多模型集成成本。
技术难点与解法
难点 1 Node.js 代理穿透
系统需走本地 HTTPS 代理（127.0.0.1:7890）访问境外 AI API。Node.js 原生模块不自动读取系统代理，且环境变量大小写（HTTPS_PROXY vs https_proxy）不一致导致静默失败。
同时检测四种写法，通过 HttpsProxyAgent 手动注入所有 HTTP 客户端
难点 2 多供应商 Token 限流与降级
三个 AI 供应商同时不可用：GROQ 100k/天耗尽、Anthropic 无余额（403）、Gemini 配额耗尽（429）。单靠任意一家无法保障稳定输出。
实现优先级降级链，Prompt 内容截断至 400 字/角色，控制单次 Token 消耗
难点 3 AWS Polly 中文语音合成
需同时指定 Engine: 'neural'、LanguageCode: 'cmn-CN'、VoiceId: 'Zhiyu' 三个参数，缺少任一参数均会回退到非中文或标准音质引擎。
明确三参数组合，流式读取 AudioStream 并拼接 Buffer 写入 MP3
难点 4 静态 Web 预构建架构
纯静态页面无法在运行时读取服务器文件。需将所有 results/*.md 预先聚合，计算 ISO 周标签（上上上周/上上周/本周），再输出为 data.json 供前端加载。
build-web.js 预构建脚本，基于周一为起点计算周次相对标签
难点 5 EC2 进程持久化
SSH 断开后，nohup 启动的 Node.js 进程随会话终止，导致服务不可访问。需要与 SSH 会话生命周期解耦的守护方式。
PM2 进程管理器 + pm2 startup systemd，实现开机自启与进程守护
难点 6 AWS SES 沙箱限制
SES 沙箱模式下，所有收件人邮箱必须单独在控制台完成验证，否则返回 554 错误。163.com 邮箱需用户手动点击验证链接。
在 SES 控制台（ap-northeast-3）逐一添加并验证收件人邮箱
