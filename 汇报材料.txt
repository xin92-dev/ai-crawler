================================================================
AI 资讯周报系统 — 技术汇报材料
================================================================

一、项目概述
----------------------------------------------------------------
本项目是一套全自动的 AI 资讯采集、分析与分发系统。
从多个 AI 领域权威平台爬取内容，借助大语言模型按角色生成
分析报告，并通过 Web 展示、邮件推送、音频播客三种形式分发。

覆盖角色：产品经理 / 开发者 / 工程师 / 运营


二、AI 相关技术与名词
----------------------------------------------------------------

【大语言模型 LLM（Large Language Model）】
  项目使用了三个 LLM 供应商：
  · GROQ — 部署了 Meta 的 llama-3.3-70b-versatile 模型
    免费额度：每日 100,000 tokens，每分钟 6,000 tokens
  · Google Gemini — gemini-2.0-flash，每日 1,500 次请求免费
  · Anthropic Claude — claude-haiku-4-5，按量付费

【Prompt Engineering（提示词工程）】
  为每个角色设计了专属的系统提示词（System Prompt），
  让同一批内容能产出面向不同受众的差异化分析。
  例如产品经理侧重"机会点和竞品动态"，
  工程师侧重"论文解读和基础设施"。

【多模型降级策略（Fallback Chain）】
  音频脚本生成采用优先级链：
  GROQ → Gemini → Anthropic
  任意一个不可用（超配额/无余额）时自动切换下一个，
  保证服务不中断。

【Token（词元）与配额管理】
  Token 是 LLM 计量单位，约 1 个中文字 = 1.5~2 tokens。
  项目遇到的核心挑战之一：GROQ 每日 100k token 上限，
  大量分析任务会快速耗尽配额，需要控制单次请求的 Prompt 长度。

【RAG 相关概念（间接涉及）】
  项目爬取的 arXiv / Papers with Code 内容包含大量
  向量数据库、Embedding、RAG 架构等前沿论文，
  这些是当前 AI 工程领域的核心技术方向。

【TTS（Text-to-Speech 文字转语音）】
  使用 AWS Polly 神经语音引擎，
  选用 Zhiyu 中文普通话神经语音（Neural Engine），
  将 LLM 生成的播客脚本合成为 MP3 音频文件。
  Neural 引擎相比标准引擎更自然流畅，接近真人语气。

【多角色 Persona 分析】
  同一份资讯，系统会用不同的角色视角各分析一遍：
  · PM 视角：产品机会、竞品动态、必读清单
  · 开发者视角：开源工具、实战教程、技术雷达
  · 工程师视角：论文解读、模型架构、基础设施
  · 运营视角：热点话题、内容日历、社区舆情

【OpenAI-Compatible API】
  GROQ 和 Gemini 均提供兼容 OpenAI 接口格式的 API，
  项目用同一套 OpenAI SDK 代码接入不同供应商，
  只需切换 baseURL 和 apiKey，降低了多模型集成成本。


三、技术难点
----------------------------------------------------------------

【难点 1：多源异构数据爬取】
  不同平台（HackerNews / arXiv / HuggingFace 等）
  的页面结构、分页逻辑、反爬策略各不相同，
  需要为每个源单独开发解析逻辑。
  部分源（Papers with Code / HuggingFace）在国内
  网络环境下频繁超时，需要代理配置。

【难点 2：Token 配额控制】
  LLM 按 Token 计费/限流，分析 4 个角色 × 3 个分析维度
  = 12 次 API 调用，每次约 2,000~4,000 tokens，
  极易触碰免费配额上限。
  解决方案：控制输入内容截断长度 + 多供应商降级兜底。

【难点 3：代理环境下的 Node.js 网络请求】
  系统需走本地 HTTPS 代理（127.0.0.1:7890）才能访问
  境外 API（GROQ / Anthropic / AWS 等）。
  Node.js 原生 https 模块不自动读取系统代理，
  需手动为每个 SDK 注入 HttpsProxyAgent，
  且环境变量大小写（HTTPS_PROXY vs https_proxy）
  也会导致代理失效。

【难点 4：多 AI 供应商统一接入】
  Anthropic SDK 与 OpenAI SDK 的 API 调用方式不同：
  · OpenAI 格式：response.choices[0].message.content
  · Anthropic 格式：response.content[0].text
  实现 Fallback Chain 时需要对两种格式分别处理。

【难点 5：AWS SES 沙箱限制】
  AWS SES 默认处于沙箱模式，只能向"已验证邮箱"发送邮件。
  若要向任意邮箱发送，需申请移出沙箱（Production Access）。
  这对新用户是一个常见的部署障碍。

【难点 6：EC2 后台进程持久化】
  通过 SSH 启动的 Node.js 进程会随 SSH 会话断开而退出。
  需使用 PM2 进程管理器守护进程，并配置开机自启（systemd），
  保证服务在 EC2 重启后自动恢复。

【难点 7：静态资源权限问题】
  Nginx 以独立用户（nginx）运行，默认无权读取
  /home/ec2-user/ 目录下的文件，
  需要手动给目录链路添加可执行权限（chmod o+x）。
  最终切换为 Express.js 直接托管，绕过了此问题。


四、系统架构简述
----------------------------------------------------------------

  [数据源] → [爬虫层] → [本地 JSON 存储]
                                ↓
                        [LLM 分析引擎]（GROQ/Gemini/Anthropic）
                                ↓
                ┌───────────────┼───────────────┐
           [Markdown 报告]  [播客音频]      [邮件推送]
                ↓               ↓
           [Web 展示]      [AWS Polly TTS]
                ↓
        [EC2 + PM2 部署]

----------------------------------------------------------------

· 支持微信/钉钉 Webhook 推送
· 用 Whisper 模型实现音频转文字（反向链路）
